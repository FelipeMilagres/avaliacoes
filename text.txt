Levantamento de Requisitos

Projeto: Software Selection – Device Farm
Versão: 1.0
Status: Em elaboração
Responsável: Centro de Excelência de Qualidade
Data: (preencher)

1. Contexto

No contexto de evolução das práticas de qualidade e automação de testes, surge a necessidade de avaliar soluções de Device Farm que possibilitem a execução de testes mobile em dispositivos reais, garantindo maior fidelidade na validação das aplicações.

Aplicações mobile em ambientes financeiros exigem alto nível de confiabilidade, compatibilidade entre dispositivos e validação em cenários reais de uso. Dessa forma, o acesso a dispositivos físicos representativos do mercado torna-se essencial para garantir a qualidade da experiência do cliente.

Considerando o contexto de um ambiente bancário com restrições de segurança e integração com infraestrutura interna, a solução selecionada deverá garantir:

execução de testes em dispositivos reais;

integração segura com ambientes internos;

compatibilidade com a esteira de automação existente;

conformidade com políticas de segurança da instituição.

Este documento apresenta o levantamento de requisitos funcionais, requisitos técnicos, riscos e restrições que deverão ser considerados no processo de seleção de uma solução de Device Farm.

2. Objetivo

O objetivo deste levantamento é definir os requisitos necessários para adoção de uma solução de Device Farm que permita:

execução de testes mobile em dispositivos reais;

execução de testes manuais e automatizados;

integração com pipelines de integração contínua;

suporte às necessidades operacionais dos times de QA.

Este documento servirá como base para:

avaliação técnica de fornecedores;

execução de provas de conceito (POC);

comparação entre soluções disponíveis no mercado.

3. Requisitos Funcionais

A solução de Device Farm selecionada deverá oferecer suporte às seguintes funcionalidades.

3.1 Execução de testes em dispositivos reais

A solução deve disponibilizar dispositivos físicos reais (Real Devices) para execução de testes mobile.

Não serão consideradas soluções baseadas exclusivamente em:

emuladores

simuladores

virtualização de dispositivos

O uso de dispositivos reais é necessário para garantir:

validação do comportamento real do sistema operacional;

análise de performance da aplicação;

validação de recursos nativos do dispositivo;

reprodução mais fiel de cenários encontrados em produção.

3.2 Execução de testes manuais

A ferramenta deve permitir que usuários executem testes manuais diretamente nos dispositivos disponíveis na plataforma.

Entre os recursos esperados estão:

acesso remoto aos dispositivos;

interação completa com a interface do dispositivo;

suporte a múltiplos usuários utilizando dispositivos simultaneamente.

3.3 Execução de testes automatizados

A solução deve suportar execução de testes automatizados em dispositivos reais.

Deve ser compatível com tecnologias de automação utilizadas pelos times, incluindo:

Appium

WebdriverIO

Além disso, deve permitir execução de testes automatizados diretamente em dispositivos físicos.

3.4 Execução paralela de testes

A solução deve permitir execução paralela de testes, possibilitando:

execução simultânea em múltiplos dispositivos;

execução paralela de testes automatizados;

uso simultâneo por diferentes usuários.

Esse recurso é essencial para reduzir o tempo de execução de testes de regressão e ampliar a cobertura de validação.

3.5 Gerenciamento e instalação de builds

A ferramenta deve permitir gerenciamento eficiente das builds utilizadas nos testes.

Entre os recursos esperados estão:

upload manual de builds (APK / IPA);

instalação de builds diretamente nos dispositivos;

suporte à distribuição de builds para diferentes dispositivos.

3.6 Captura de evidências

Durante a execução dos testes, a solução deve permitir o registro de evidências.

Entre os recursos esperados estão:

captura de screenshots

gravação de sessões

exportação de evidências para análise posterior.

3.7 Observabilidade e debugging

A solução deve oferecer recursos que facilitem a análise de problemas durante a execução dos testes.

Entre os recursos esperados estão:

monitoramento de tráfego de rede;

acesso a logs do dispositivo;

inspeção de elementos da interface;

acesso a logs de execução.

4. Requisitos de Integração

A solução deve permitir integração com os principais componentes da esteira de desenvolvimento e automação utilizada pelos times.

4.1 Integração com GitLab CI/CD

A ferramenta deve permitir integração com pipelines GitLab CI/CD, possibilitando:

execução de testes automatizados diretamente nos pipelines;

execução de testes em dispositivos reais durante o processo de build;

coleta e publicação de resultados de execução.

Essa integração deve permitir que os testes mobile façam parte do processo de integração contínua.

4.2 APIs de integração

A solução deve disponibilizar APIs ou interfaces de automação que permitam:

provisionamento de dispositivos;

execução de testes;

coleta de resultados;

integração com pipelines de CI/CD.

5. Requisitos de Infraestrutura e Segurança

Considerando o contexto de um ambiente bancário, a solução deve atender aos seguintes requisitos.

5.1 Acesso à rede interna

A ferramenta deve permitir acesso seguro aos ambientes internos do banco.

Entre os mecanismos esperados estão:

integração via VPN

utilização de túnel seguro

comunicação segura com ambientes internos (DEV / QA / HML).

5.2 Conformidade com políticas de segurança

A solução deve atender às políticas de segurança da informação adotadas pela instituição, incluindo:

criptografia de comunicação;

controle de acesso por usuário;

rastreabilidade de atividades;

segregação de ambientes.

5.3 Isolamento de sessões

A ferramenta deve garantir isolamento entre sessões de teste, incluindo:

limpeza automática de dados após utilização do dispositivo;

não persistência de dados sensíveis;

reinicialização dos dispositivos entre sessões.

6. Requisitos de Performance

A solução deve apresentar níveis adequados de performance para garantir eficiência na execução dos testes.

Entre os requisitos esperados estão:

baixa latência na interação com dispositivos;

estabilidade das sessões de teste;

tempo reduzido para instalação de builds;

alta disponibilidade de dispositivos.

7. Riscos Identificados

Durante o processo de avaliação da solução, alguns riscos técnicos devem ser considerados.

Entre os principais riscos identificados estão:

impossibilidade de acesso aos ambientes internos do banco;

limitações de integração com a esteira de automação;

indisponibilidade de modelos específicos de dispositivos;

problemas de performance ou latência;

restrições de segurança que impeçam o uso da solução.

8. Restrições

Algumas restrições devem ser consideradas no processo de seleção da solução.

Entre elas:

conformidade com políticas de segurança da instituição;

necessidade de integração com ambientes internos;

aderência às ferramentas atualmente utilizadas na esteira de testes;

adequação ao orçamento disponível.

9. Resultado Esperado

Ao final deste levantamento espera-se:

definição clara dos requisitos técnicos da solução de Device Farm;

validação da viabilidade de integração com a infraestrutura do banco;

identificação de riscos e restrições;

apoio à avaliação e comparação entre fornecedores no processo de Software Selection.
